# Architecture Audit — V1 vs V2 (2026-07-13)

Read-only forensic comparison of `D:/mlb_model` (V1) and `D:/mlb_model_v2` (V2) to decide
which project is the stronger **foundation** for a long-term MLB analytics/betting platform.
No files in either project were modified. This is the rationale behind the decisions encoded
in [`../CLAUDE.md`](../CLAUDE.md).

## Question asked

Not "which is more built out?" but "which has the better **framework** to build onto — the way
it pulls stats, whether that's sensible, efficient, and complete?" Feature count, dashboards,
and existing predictions were explicitly excluded from judgment.

## Verdict

**Build on V2. Freeze V1 as a read-only parts donor.**

V1 is more built out (player props, totals/run-line, line-movement, a Streamlit dashboard);
V2 is better engineered. Since only the engineering foundation was in scope, V2 wins clearly.
The one caveat: V2 is moneyline-only today, so proceeding on V2 means *porting* a few modules
from V1 — not discarding V1's ideas.

## Data source map

- **V1**: MLB Stats API (primary), Retrosheet GitHub CDN (fallback), Baseball Reference via
  pybaseball (last resort, "may be blocked" per its own comments), Baseball Savant, The Odds
  API, Stathead (scraped). The `01_collect_data.py` header reads as a changelog of *reactive*
  fixes — the source strategy grew by patching around blocks.
- **V2**: **`baseballr` as a unified primary layer** — MLB Stats API, FanGraphs, and Statcast
  all pulled through it, with honest documented fallbacks (e.g. `04_collect_statcast_team.R`
  explains why it uses FanGraphs' Statcast-derived metrics instead of a flaky direct Savant
  scrape). This is the baseballr-centric design that was the whole point of V2.

## Data lineage & leakage discipline (decisive for a betting model)

V2 has a clean, traceable, leakage-aware path:
- `baseballr` → `data/raw/*.csv` (raw preserved, incremental) → `features/builder.py` →
  `data/processed/feature_matrix.csv` → train/predict.
- `features/builder.py` is a **single source of truth imported by both training and
  prediction**, so they can't silently diverge (train/serve skew — the classic betting bug).
- Every rolling feature uses `.shift(1)` so a game never sees its own result. Odds are captured
  as dated `odds_close_*` snapshots for real closing-line-value tracking. Raw data is preserved
  via probables/lineups caches and per-day odds snapshots.

V1's lineage exists but is harder to trust: feature logic is spread across
`03_feature_engineering.py` (757 lines) + `09_advanced_features.py` (992 lines), and prediction
is a *separate* 1,477-line `06_predict_today.py` — so train and predict logic can drift.

## Technical-debt evidence in V1

- `savant_stats_fetch.py` and `savant_stats_fetch_v4.py` are **byte-for-byte identical**
  (v2/v3 near-dupes) — four copies of one fetcher.
- A **complete second project nested at `MLB/`** with its own backend/frontend/db and duplicated
  `04/06/10/12` scripts — two sources of truth for the same model.
- ~22 one-off `check_*/fix_*/debug_*/show_*` scripts committed at the repo root.

## Migration plan (V2 as backbone)

1. Freeze V1 read-only; use it only as a donor.
2. Keep building on V2's spine (`R/` collectors → `features/builder.py` → scripts) — it is
   already the "clean layered" target architecture.
3. Port from V1 one clean module at a time, into V2's structure:
   - Player props (V1 `08/2x_*`) → new `R/` collectors + `features/props_builder.py` mirroring
     the moneyline pattern.
   - Dashboard (V1 `app.py`) → point at V2 outputs.
   - Line-movement/CLV — fold V1 `11_line_movement.py` ideas into existing
     `07_capture_closing_lines.py`.
4. Do NOT port: the nested `MLB/` project, the 4 savant fetchers, any root `check_/fix_/debug_`
   script.
5. Adopt light governance only: a one-page source registry, a leakage/feature checklist, a
   running tech-debt list. Skip ADRs and per-dataset data dictionaries until a collaborator joins.

## Open item

Whether all 13 R collectors currently parse/run clean was **not** verified (audit was read-only,
and the smoke driver only tests the Python side). Parse-check `R/` with Rscript before relying on
V2 as the sole backbone.
